# Stage 9 Topics — 意识体深度演化

Stage 8 完成了双层意识体的基础架构：守护意识（调度）+ 领域子意识（执行）。
但当前守护意识仅起分流作用，尚未充分发挥 context 管理的潜力。

以下话题逐个讨论解决。

---

## 类比框架：从作坊到产业

旧的 Agent 就像是**小作坊**，一个人什么活都干，应对不了大的业务。

Tramber 的演进路径对应企业组织形态的升级：

```
小作坊（旧 Agent）                    产业生态（目标）
一个师傅干所有事                       专业分工、高效协作
工具调用 + 单循环                      分层管理、资源调度
context 无限膨胀 → 失控               context 按需分配 → 可控
```

### 组织映射

| Tramber 概念 | 企业类比 | 职责 |
|-------------|---------|------|
| 用户 | **董事会** | 制定战略方向，下达指令，审批重大决策 |
| 守护意识（Guardian） | **公司管理层** | 代表董事会利益，提纲挈领，高瞻远瞩。掌握全局视野，调度资源，不亲自干活 |
| 执行意识（Domain Child） | **具体部门**（研发部、文档部、运维部） | 像锋利的剑，指哪打哪，敢打敢冲。专注领域，高效执行 |
| Scene | **专业公司** | 一家公司专门解决对应领域的问题（编码公司、文档公司、部署公司） |
| 多个 Agent | **Partners**（合作伙伴） | 多个 agent 组成 partners，高效协作 |
| Agent Network | **产业链** | 更多产业公司聚在一起，形成产业链 |

### 管理层（守护意识）的工作方式

- **不亲自搬砖**，但要知道每个部门在干什么、为什么干、干得怎么样
- 维护**全局视野**：公司整体战略目标、各部门进度、关键决策、资源状况
- 做调度决策：什么任务分给什么部门、部门之间如何协调
- 向董事会（用户）汇报：不转发部门的原始报告，而是用自己的理解做高管摘要

### 执行部门（领域子意识）的工作方式

- **专注高效**：只关心自己领域的事，不管其他部门
- **结果导向**：拿到任务就执行，不犹豫不越权
- **资源管理**：部门内部有自己的工作手册（执行纲领）和工具库（资源索引）
- **向上汇报**：超出领域的事不硬撑，上升给管理层
- **保持锐利**：context 不能被冗余信息拖慢，部门的工作台要始终整洁

### 演进路线

```
Stage 8（当前）             Stage 9（本次）           Stage 10+（未来）
─────────────              ──────────────           ────────────────
单个专业公司内部优化         公司间协作（Partners）     产业链（Agent Network）
├─ 管理层有全局视野          ├─ 多个 Agent 协作        ├─ Agent 间通信协议
├─ 部门有执行纲领            ├─ 跨公司任务分配          ├─ 行业标准 scene
└─ context 按需分配          └─ 共享资源池             └─ 去中心化调度
```

以下三个 Topic 的解决方案思路，基于这个类比框架展开。

## Topic 1: 守护意识的会话纲领

### 问题

守护意识当前只记录每轮分析总结（`用户：xxx → 子意识：xxx`），这是时间线式的流水记录。
随着对话增长，流水记录线性膨胀，缺少**全局视野** — 一个概括当前会话整体目标的"纲领"。

**需要**：守护意识应维护一个会话纲领（session agenda），包含：
- 会话的总体目标
- 已完成的事项
- 当前进行的事项
- 待解决的事项
- 关键决策和约束

**风险**：会话纲领在长期演进中容易丢失某些重要分支 — 纲领是对流水的压缩，压缩必然有损。
如何在保持纲领简洁的同时，不丢失重要信息？

### 已确定方向

- 纲领中每一条都必须有会话 ID 支撑，不能虚空发挥
- 每轮分析总结分配 5 位 36 进制 ID（如 `a3x7f`），作为会话事实的唯一标识
- 纲领 + 会话记录两层互相印证，ID 是锚点

### 待选方案：A + B + C 组合

#### 方案 A：类型前缀 ID（基础标识层）

给 ID 加类型前缀，一眼识别实体类别：

```
[t:a3x7f] 任务 - 查看 demos 目录
[d:b8k2m] 决策 - 技术栈选定 Three.js
[r:c4m9p] 资源 - demos/3d-beach.html（海滩场景模板）
[c:d5n3q] 约束 - 所有文件使用 UTF-8
[e:f7p1r] 事件 - 编码领域创建
```

纲领引用时：`战略方向 [t:a3x7f] [t:c4m9p]` → 能区分引用的是任务还是决策。
最轻量，LLM 只需加前缀。

#### 方案 B：实体关系图（关系层）— JSON + 内存实现

每个要素是图中的一个节点，节点之间有显式边。
**实现方式：JSON 文件 + 内存遍历**，不依赖外部图数据库或向量数据库。

实体存储（复用已有 MemoryStore，每个实体一条 JSON）：

```json
{
  "id": "t:a3x7f",
  "type": "task",
  "content": "查看 demos 目录",
  "domain": "编码",
  "relations": [
    { "type": "produces", "target": "r:b8k2m" },
    { "type": "triggers", "target": "d:c4m9p" },
    { "type": "leads_to", "target": "t:d5n3q" }
  ]
}
```

关系追踪（内存遍历，百级实体无需图数据库）：

```typescript
// 找 t:a3x7f 产生的所有资源
entity.relations.filter(r => r.type === 'produces').map(r => store.get(r.target))
```

语义检索可选路径：
1. **LLM 匹配**（当前可用）：把实体摘要列表给 LLM 判断相关性
2. **嵌入向量**（未来可加）：用 `transformers.js` 本地计算 embedding，余弦相似度检索，零外部依赖

```
[t:a3x7f] 查看 demos 目录
  ├── produces → [r:b8k2m] demos 目录结构（8个3D + 2个可视化）
  ├── triggers → [d:c4m9p] 选定 Three.js 技术栈
  └── leads_to → [t:d5n3q] 生成新的 3D 场景
```

纲领是这个图的一个视图。关系显式，因果链可追溯。
符合零外部依赖原则，百级实体规模下 JSON + 内存足够。

#### 方案 C：结构化条目（内容层）

每轮分析不再是纯文本，而是结构化 JSON：

```json
{
  "turnId": "a3x7f",
  "taskRef": "t:001",
  "domain": "编码",
  "facts": [
    { "type": "discovery", "content": "demos 包含 10 个 HTML 文件", "resources": ["r:010"] },
    { "type": "decision", "content": "Three.js 用于 3D 场景", "id": "d:020" }
  ],
  "status": "completed"
}
```

纲领从这个结构化数据**由代码生成**，不是 LLM 自由发挥。
LLM 负责提取 facts，代码负责组装纲领 → 防失真。

#### 组合方案：A + C 做基础，B 做关系增强

1. 用**类型前缀 ID**（A）做基础标识
2. 每轮要求 LLM 输出**结构化 facts**（C），而非纯文本
3. 实体之间维护**显式关系**（B），因果链可追溯
4. 纲领由代码从 facts + 关系推导生成，LLM 只提取不编造

#### 方案对比

| 方案 | 离散程度 | LLM 负担 | 实现复杂度 | 防失真 |
|------|---------|---------|-----------|--------|
| A 类型前缀 ID | 中 | 低 | 低 | 中 |
| B 实体关系图 | 高 | 高 | 高 | 高 |
| C 结构化条目 | 高 | 中 | 中 | 高 |
| A+B+C 组合 | 最高 | 中高 | 中高 | 最高 |

## Topic 2: 执行意识体的 Context 自组装

### 问题

当前执行意识体是传统 agent 流程 — conversation 不断追加消息。
当 context 增大到阈值时需要压缩（摘要/截断），这导致：
- **慢**：压缩需要额外的 LLM 调用
- **信息丢失**：摘要必然丢细节，之前积累的任务理解、规则、文件内容被稀释
- **质量持续下降**：每次压缩都不可逆地损失信息

传统 agent 的 context 是**消耗品**——越用越旧，压缩一次差一次。

### 核心认知：共享 Memory + 自主组装

A+B+C 方案（Topic 1）建立的实体图是**共享基础设施**，守护意识和执行意识都读写同一套数据。

执行意识的 context 不是由守护意识"分配"的，而是**执行意识自己从共享 Memory 组装**：

```
                ┌─────────────────────────┐
                │   共享 Memory（A+B+C）    │
                │                         │
                │  实体：t: d: r: c: e:    │
                │  关系：produces, leads_to │
                │  结构化 facts            │
                └─────┬──────────┬────────┘
                      │          │
              ┌───────▼──┐  ┌───▼────────┐
              │ 守护意识  │  │ 执行意识    │
              │          │  │             │
              │ 读→生成纲领│  │ 读→组装context│
              │ 写→调度决策│  │ 写→任务发现  │
              └──────────┘  │ 写→新资源    │
                            └─────────────┘
```

### 自组装流程

执行意识收到任务 `[t:d5n3q]` 时，从共享 Memory 查询关联实体，自动组装 context：

```
执行意识收到任务 [t:d5n3q] 生成新 3D 场景
  │
  ▼ 从共享 Memory 查询关联实体
  │
  t:d5n3q.requires → r:e4p1k（3d-beach.html 模板）   ← 自动关联
  t:d5n3q.root_task → t:a3x7f（查看 demos 目录）      ← 追溯上游
  t:a3x7f.triggers → d:c4m9p（选定 Three.js）         ← 继承决策
  t:d5n3q.blocked_by → c:f7q2w（需确认场景类型）       ← 约束检查
  │
  ▼ 自动组装 context
  │
  ## 执行纲领
  任务 [t:d5n3q] 生成新 3D 场景
  上游 [t:a3x7f] 已完成探索
  技术决策 [d:c4m9p] Three.js 0.160.0
  模板资源 [r:e4p1k] 3d-beach.html（如需细节，recall）
  约束 [c:f7q2w] 已解除

  ## 最近对话
  （最后 N 轮）
```

**关联自动组装，不够再去获取**：
- `requires` → 自动拉入相关资源摘要
- `triggers` → 自动继承上游决策
- `blocked_by` → 自动检查约束状态
- 资源只有摘要，需要完整内容时 → `recall_resource(r:e4p1k)`

### Context 膨胀时的行为：重建而非压缩

```
执行意识 context 接近阈值
  │
  ▼ 不是压缩，而是重建
  │
  1. 丢弃旧的对话消息
  2. 从共享 Memory 重新查询实体图
  3. 按关联自动组装新的 context
  4. 资源只保留摘要索引，细节按需 recall
  │
  ▼ 执行意识继续工作，质量恒定
```

重建是**无损的**——因为实体是离散的、有 ID 的、关系是显式的，重新组装和第一次组装的结果一致。

```
传统：context → 膨胀 → 压缩 → 丢信息 → 再膨胀 → 再压缩 → 质量持续下降

A+B+C：context → 膨胀 → 重建 → 从实体图无损组装 → 质量恒定
```

### 两个意识层的读写关系

| 操作 | 守护意识 | 执行意识 |
|------|---------|---------|
| **读** | 全量实体图 → 生成会话纲领 | 本领域实体图 → 组装执行 context |
| **写** | 领域切换决策、路由策略、约束提取 | 任务发现、文件资源、技术决策、执行进度 |
| **触发重建** | 每轮更新纲领 | context 接近阈值时重建 |

执行意识产生的新发现（读了什么文件、发现了什么结构、做了什么修改）都写回共享 Memory，带类型 ID 和关系边。守护意识立刻可见，纳入纲领更新。

### 与 Topic 1 的关系

Topic 1 和 Topic 2 是**同一套 A+B+C 的两个 view**：
- Topic 1：守护意识读实体图 → 生成会话纲领（全局视野）
- Topic 2：执行意识读实体图 → 组装执行 context（领域视野）

底层同一套数据，上层各自组装所需视图。一次设计，两个问题同时解决。

## Topic 3: 执行意识体的纲领与资源管理

### 问题

在 Topic 2 的自组装基础上，执行意识体需要解决两个具体问题：

1. **执行纲领不失真** — 执行纲领从哪来、怎么更新、怎么保证准确？
2. **资源高效调用** — 执行中遇到的信息怎么分类保存、按需取回？

### 执行纲领（从共享 Memory 推导，不是独立维护）

执行纲领不是另一套独立的结构，而是**共享 Memory 中本领域实体的一个视图**：

```
守护意识的 view（会话纲领）：
  全量实体 → 战略方向 + 任务全景 + 关键决策

执行意识的 view（执行纲领）：
  本领域实体 → 当前任务 + 关联资源 + 约束条件
```

不需要"同步"问题——因为只有一份数据，两个 view 实时一致。

执行纲领的更新 = 执行意识写新实体到共享 Memory → 自动反映在下次组装的 context 中。

### 资源管理（A+B+C 中的 r: 类型实体）

执行意识在执行中遇到的信息分类保存为资源实体：

```
[r:e4p1k] 资源
  type: file
  path: demos/3d-beach.html
  summary: Three.js 海滩场景，含海洋、沙滩、椰子树，~13.9KB
  relations:
    - { type: "produced_by", target: "t:a3x7f" }    ← 由哪个任务产出
    - { type: "required_by", target: "t:d5n3q" }     ← 被哪个任务依赖
```

```
[r:f5q2k] 资源
  type: knowledge
  content: Three.js 场景使用 OrbitControls 做交互，importmap 引入 CDN 模块
  relations:
    - { type: "discovered_in", target: "t:a3x7f" }   ← 在哪个任务中发现
```

**概括粒度**：资源实体存摘要，原始内容存在文件系统（文件本身不动）。
调用时 `recall_resource(r:e4p1)` → 从摘要定位文件 → `read_file` 获取完整内容。

### 资源按需调用的流程

```
执行意识组装 context 时：
  ┌─ 执行纲领（从实体图自动推导）
  ├─ 资源索引（关联资源的摘要列表）
  └─ 最近 N 轮对话

执行过程中需要某个资源详情：
  → recall_resource(r:e4p1)
  → 系统从摘要中找到 path: demos/3d-beach.html
  → read_file 获取完整内容
  → 临时加入当前对话
  → 完成后从对话中移除，不影响 context 大小
```

### 核心挑战的解决状态

| 挑战 | 解决思路 | 状态 |
|------|---------|------|
| 执行纲领不失真 | 纲领 = 共享 Memory 的视图，不是独立维护 | ✅ Topic 2 已定 |
| 高效调用资源 | 资源实体存摘要，recall 时再取详情 | 待细化 |
| 主从共享 | 同一套 Memory，不存在同步问题 | ✅ Topic 2 已定 |
| 资源分类 | `r:` 类型实体，子类 file/knowledge/api/pattern | 待细化 |
| 资源粒度 | 摘要 + 关联路径，按需 fetch 完整内容 | 待细化 |

### 待讨论

- 资源实体的子类型：file、knowledge、api、pattern 是否足够？
- 资源摘要由谁生成：LLM 提取还是代码自动截取？
- `recall_resource` 的实现：是新虚拟工具还是复用 recall_memory？
- 执行意识写回资源的时机：每次工具调用后？还是任务完成时？
- 资源去重：同一文件被多次读取时如何合并？
